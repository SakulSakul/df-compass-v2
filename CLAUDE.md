# CLAUDE.md — DF COMPASS v2

AI(Claude Code)가 이 저장소에서 작업할 때 따르는 원칙.
**설계 기준은 DESIGN.md 단일 문서다. 모든 구현 판단은 그 문서의 §0 품질 서열에
종속되며, 서열과 충돌하는 최적화·기능은 만들지 않는다.**

## Part 0 — 부트스트랩 (매 세션 시작 시 무조건 실행)

사용자가 무엇을 말하든("시작해", "이어서 해", "고") 다음 순서로 스스로 진행한다:

1. `STATUS.md`를 읽고 **현재 Phase**와 미완 체크리스트를 파악한다.
2. `docs/phases/phase-{N}.md`(현재 Phase의 작업 지시서)를 읽는다.
3. 지시서의 미완 항목 중 **가장 앞의 것부터** 진행한다. 항목 완료 시마다
   `STATUS.md`의 체크박스를 갱신하고 완료 근거(실행 출력 요약)를 한 줄 남긴다.
4. `[사용자 액션]` 표시가 붙은 항목은 코드로 대신하지 말고, 사용자에게 정확히
   무엇을 해야 하는지(어느 파일을 어디서 실행하고 어떤 결과를 알려달라)를 안내하고
   그 항목을 건너뛴 채 진행 가능한 다음 항목으로 넘어간다. 진행 가능한 항목이
   없으면 거기서 멈추고 대기한다.
5. Phase 종료 조건이 전부 충족되면 **다음 Phase를 시작하지 말고** STATUS.md의
   Phase 판정 섹션을 채운 뒤 사용자에게 판정 확인을 요청한다. 사용자가 승인해야만
   STATUS.md의 현재 Phase를 올리고 다음 지시서로 넘어간다.

## Part 1 — Karpathy 4 Principles

1. **Think Before Coding** — 가정을 명시하고, 해석이 갈리면 묻고, 더 단순한 길이 있으면 말하라.
2. **Simplicity First** — 요청 밖 기능·단일 사용 추상화·미래 대비 유연성 금지.
3. **Surgical Changes** — 요청에 추적되는 최소 변경만. 인접 코드 개선 금지.
4. **Goal-Driven Execution** — 작업을 검증 가능한 목표로 변환하고 검증까지 돌려라.

## Part 2 — v2 전용 규율

### A. Phase 규율 (가장 중요)
- **현재 Phase 밖의 코드를 작성하지 않는다.** 다음 Phase가 "어차피 필요"해 보여도
  선구현 금지. Phase 전환은 Part 0-5의 사용자 승인으로만.
- Phase 0 타임박스 2주. 초과 조짐이면 스코프 축소를 제안하라.
- CAG 관련 코드는 Phase 2 전에 한 줄도 만들지 않는다 (게이트 G1·G2·G3 선행).

### B. eval-first (DESIGN.md ADR-6)
- retrieval·prompt·boost 규칙을 바꾸는 모든 변경은 eval diff를 먼저 보여준다.
- eval 하니스 없는 상태에서 검색 품질을 "좋아졌다"고 주장하지 않는다.
- 검색 수준 eval = 상시, 인용 수준 eval = 병합 전 (§8 2단화).

### C. 조항 정규화 단일 함수 (§3)
- "제N조·제N조의M" 파싱 함수는 레포에 **정확히 하나**(`compass_engine/articles.py`).
  파서·적재기·검증기·인용 추출기가 전부 이걸 import. 복제·인라인 정규식 우회 금지.

### D. 결정론 우선
- 인용 검증(§5)·critical 게이트·answer_guard는 LLM 프롬프트 준수가 아니라
  코드/DB 대조로 보증한다. "프롬프트에 지시했으니 됐다"는 완료가 아니다.

### E. silent fail 금지 (v1 17시간 사고의 교훈)
```python
# ❌ except Exception: pass
# ✅ trace 기록 + stderr + 회로 상태 반영
```
- 새 컬럼을 .select()에 넣기 전 information_schema로 물리 컬럼 실재 확인
  (RPC 반환 키 ≠ 물리 컬럼).

### F. 배포 상태 3층 보고
소스/커밋 · 배포 트리거 · 실제 앱 로딩을 분리 보고. 미확인 레이어는 명시적 갭으로.

### G. 검증 없는 완료 선언 금지
명령을 실제로 돌린 출력과 함께 보고하거나, `(추론: 미실행)`을 명시하라.
STATUS.md 체크는 이 기준을 충족한 항목에만 한다.

### H. 심의 대상 변경은 구현 전 정지
다음에 해당하는 변경은 코드를 작성하기 전에 멈추고, "council 심의 대상"으로
사용자에게 플래그한다 (사용자가 별도 채널에서 심의 후 결과를 지시로 준다):
- 엔진 계약·DB 스키마·§5 검증기 로직 등 아키텍처 근간 변경
- critical 경로·핫라인·안전 게이트를 지나는 모든 변경
- Phase 전환, v1 컷오버, 신규 외부 의존(벤더·API) 도입
- eval로 회귀를 검출할 수 없는 종류의 변경
반대로 eval 게이트가 조건으로 걸린 가역적 변경(모델 env·프롬프트·boost 규칙)은
기존 절차대로 진행한다 — 심의를 요식으로 만들지 않는다.
